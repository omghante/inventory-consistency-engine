## Key Design Decisions

1. **Per-product StampedLock** вҖ” Different products process in parallel with zero cross-product contention. Write operations (merge) acquire write locks. Read operations (getStock) use optimistic reads for zero-contention queries under flash sale conditions.

2. **Lock-free statistics (AtomicLong)** вҖ” Engine-level counters (totalMerges, appliedCount, rejectedCount, etc.) use AtomicLong for lock-free thread-safe accumulation. No counter increments are lost under concurrent load.

3. **Metadata-driven degradation** вҖ” Same merge() call, different behavior based on available metadata. System never crashes when upstream services are unavailable.

4. **Bounded vector clocks** вҖ” Supports configurable max-entries pruning to prevent unbounded memory growth in systems with many warehouse nodes. Retains the most causally significant entries (highest counters). Follows Amazon Dynamo's clock pruning strategy.

5. **Append-only event log** вҖ” Enables replay without data loss. Combined with snapshot-based fast recovery.

6. **Zero external dependencies** вҖ” Pure Java, self-contained.

## Concurrency Model

```
Thread 1 (PS5 order)  в”Җв”ҖвҶ’ StampedLock[PS5].writeLock()  в”Җв”ҖвҶ’ merge pipeline
Thread 2 (PS5 order)  в”Җв”ҖвҶ’ StampedLock[PS5].writeLock()  в”Җв”ҖвҶ’ waits (same product)
Thread 3 (XBOX order) в”Җв”ҖвҶ’ StampedLock[XBOX].writeLock() в”Җв”ҖвҶ’ merge pipeline (parallel)
Thread 4 (PS5 query)  в”Җв”ҖвҶ’ StampedLock[PS5].tryOptimisticRead() в”Җв”ҖвҶ’ returns immediately
```

- **Writes** to the same product are serialized (required for correctness)
- **Writes** to different products run in parallel (no shared state)
- **Reads** use optimistic locking вҖ” zero contention unless a concurrent write invalidates the stamp, in which case it falls back to a read lock
- **Statistics** are lock-free AtomicLong вҖ” no mutex required for counters
